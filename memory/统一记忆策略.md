## 统一检索顺序
1. **session JSONL**：原始事实源，优先查跨 session 历史
2. **LCM**：查被压缩过的旧对话摘要
3. **memory 文件**：查长期规则、复盘、偏好、项目结论
4. **ontology**：查结构化实体/关系（人、账号、项目、任务、偏好、文档）

## 各层职责

### 1. Session JSONL
适合存：
- 原始对话
- 工具调用过程
- 某次任务的完整上下文

不适合存：
- 需要长期稳定复用的规则结论

### 2. memory/*.md
适合存：
- 日记/复盘
- 踩坑总结
- 已确认的流程规则
- 每日重要事件

### 3. memory/hot / warm
- HOT：当前进行中的高优先级事项、近期风险、当前操作原则
- WARM：稳定偏好、常用配置、长期有效规则

### 4. ontology
适合存：
- Person / Agent / Account / Project / Task / Preference / Document / Session
- 谁负责什么
- 哪个账号对应哪个 agent
- 哪个项目在哪些 agent/session 做过
- 某个文档/工作流属于哪个项目

## 写入规则
### 应写入 ontology 的内容
- Gary、各 Agent、各账号身份
- 项目与任务
- agent 分工关系
- 账号与 agent 的映射
- 稳定偏好与规则
- 关键文档/关键工作流

### 应写入 memory 文件的内容
- 每日完成事项
- 问题与修复记录
- 经验总结
- 需要人类可读的 SOP / 复盘

### 不要只留在聊天里的内容
- 已确认的偏好
- 已跑通的工作流
- 关键权限/配置问题
- 经常会被再次问到的结论

## 回答“之前做过什么/记得吗”时的 SOP
1. 先查相关 agent/session JSONL
2. 再查 LCM
3. 再查 memory/*.md
4. 最后查 ontology 做结构化补全
5. 汇总后明确说明：事实来自哪里、哪些是推断
