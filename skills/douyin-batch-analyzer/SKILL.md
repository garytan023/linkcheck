# 抖音批量素材分析

按下面流程执行，不要跳步。

## 目标

输入一个或多个抖音分享链接，产出：
1. 视频本地文件
2. 音频文件
3. 中文转写文本
4. 基于转写内容的结构化分析
5. 飞书表格或 Markdown 汇总

## 正确工作流

### 1. 标准化输入
优先接收数组形式的 `share_links`。即使用户只给 1 条，也按批量流程处理。

示例：

```json
{
  "share_links": [
    "https://v.douyin.com/xxxx/",
    "https://v.douyin.com/yyyy/"
  ],
  "output_mode": "feishu_sheet",
  "template_mode": "analysis_table",
  "batch_name": "某品牌首批素材分析"
}
```

如果用户给的是分享文案而不是裸链接，先从文本里提取真实抖音短链。

---

### 2. 解析每条链接
对每个 link 单独处理，保留独立状态。

优先做法：
- 通过 `mcporter` / Douyin MCP 获取：
  - `video_id`
  - `title`
  - `download_url`

如果失败：
- 标记该条 `status=failed`
- 保留失败原因
- 继续处理下一条，不要整批中断

---

### 3. 下载视频
先把视频落盘，再做后续分析。

建议命名：
- `videos/{video_id}.mp4`

推荐用 `exec` 调 `curl` 下载。

不要一开始就尝试把视频直接发飞书；先把本地链路跑通。

---

### 4. 提取音频
用 `ffmpeg` 从 mp4 提取 mp3。

建议命名：
- `audio/{video_id}.mp3`

---

### 5. 本地 Whisper 转写
默认用本地 `whisper-cli`，不要默认依赖远程 ASR API。

优先级：
1. `~/.whisper/ggml-medium.bin`
2. `~/.whisper/ggml-base.bin`
3. 其他可用 whisper.cpp 模型

推荐命令形态：

```bash
whisper-cli -m ~/.whisper/ggml-medium.bin <audio_path> --language Chinese
```

规则：
- 先检查 `~/.whisper/` 里是否已有模型，不要重复下载
- `medium` 优先，`base` 作为回退
- 如果 `medium` 损坏或加载失败，再退回 `base`
- 不要把 MCP 自带文本提取当主链路；它可以试，但失败后应直接切到本地 Whisper

---

### 6. 拆分“事实字段”与“分析字段”
所有输出必须区分这两类。

#### 事实字段（来自真实转写/视频信息）
例如：
- 视频名称
- 总时长
- 关键情节
- 产品出现场景
- 口播原句片段
- 节日/家庭/产品/场景元素

#### 分析字段（基于事实判断）
例如：
- 视频类型
- 目标受众
- 整体节奏
- 区前卡点评价
- 产品展示话术
- 卖点利益点话术
- 卖点利益点
- 备注

必须显式说明：
- 事实字段来自转写/视频
- 分析字段来自内容判断
- 不要假装所有内容都是逐字原话

---

### 7. 模板优先
如果用户给了参考截图或指定字段，按模板输出。

默认模板字段：
- 视频名称
- 总时长
- 视频类型
- 目标受众
- 整体节奏
- 区前卡点评价
- 产品展示话术
- 卖点利益点话术
- 卖点利益点
- 备注

如果是批量分析，把每条视频作为一行汇总到总表。

推荐总表字段：
- 链接
- 视频名称
- 总时长
- 视频类型
- 目标受众
- 整体节奏
- 区前卡点评价
- 产品展示话术
- 卖点利益点话术
- 卖点利益点
- 备注
- 状态

---

### 8. 飞书输出
优先用 `feishu_sheet`，一次性创建总表。

规则：
- 先把所有记录汇总成二维数组
- 再一次性 `create`
- 不要一条视频建一张表，除非用户明确要求单条单表

默认输出：
- 先返回简短文字结论
- 再附飞书表格链接

---

## 推荐输出结构

每条视频至少整理成：

```json
{
  "source_link": "",
  "video_id": "",
  "video_title": "",
  "duration": "",
  "video_path": "",
  "audio_path": "",
  "transcript_path": "",
  "facts": {
    "关键情节": [],
    "产品出现场景": [],
    "原话片段": []
  },
  "analysis": {
    "视频类型": "",
    "目标受众": "",
    "整体节奏": "",
    "区前卡点评价": "",
    "产品展示话术": "",
    "卖点利益点话术": "",
    "卖点利益点": "",
    "备注": ""
  },
  "status": "success"
}
```

失败条目示例：

```json
{
  "source_link": "",
  "status": "failed",
  "error": "未找到有效分享链接"
}
```

## 判断准则

### 什么叫“正确做法”
- 先拿到真实转写，再做分析
- 先完成本地视频/音频/文本闭环，再输出飞书
- 批量任务允许部分成功，不能因为一条挂掉整批失败
- 明确区分事实与判断
- 输出字段优先服从用户模板，不自作聪明改结构

### 什么是错误做法
- 没转写就直接脑补内容
- 把“主题榜/热度榜”误当成用户截图模板
- 一边解析一边频繁写飞书，导致流程脆弱
- 只返回结论，不保留失败项和失败原因

---

## 批量处理规则

### 循环规则
对 `share_links` 做循环处理：

```text
for link in share_links:
  parse -> download -> extract audio -> transcribe -> facts -> analysis -> append result
```

### 容错规则
- 单条失败不阻断全局
- 最终总表保留 `状态` 列
- 失败项写明原因

### 命名规则
用 `video_id` 作为主键，避免文件覆盖：
- `videos/{video_id}.mp4`
- `audio/{video_id}.mp3`
- `transcripts/{video_id}.txt`

---

## 交付文案模板

### 简版结论

```markdown
本批共处理 X 条抖音视频：成功 X 条，失败 X 条。

整体看，素材主要集中在：
- xxx
- xxx
- xxx

飞书表格：<url>
```

### 单条结论

```markdown
这条视频本质上是“xxx + xxx + xxx”的内容结构，适合 xxx，不属于 xxx 型素材。
```

---

## 本 skill 的首选工具组合

- `exec`：调用 curl / ffmpeg / whisper-cli / mcporter
- `feishu_sheet`：创建总表
- 必要时读用户提供的截图或上下文模板

不把以下方式当默认主链路：
- 远程 ASR API
- 平台自带文本提取接口
- 直接先发飞书视频再回头分析

---

## 一句话 SOP

**多个抖音链接输入 → 循环解析下载 → 本地抽音频 → Whisper 转写 → 按模板拆事实和分析字段 → 汇总输出飞书总表。**
